## 0. Indeterminate variant (added 2026-07-21)

Added `Indeterminate` as a third value on the `Checked` variant property (previously `False`/`True` only - now `False` / `True` / `Indeterminate`). Built 6 new components: **Default color only**, Small + Medium, أ— Enabled/Hovered/Focused (no Disabled - see gap note below).

- **Thumb centering:** unlike the existing Off/On components (which position the knob with a manually-computed absolute `x` offset inside a `layoutMode: NONE` frame), the new Indeterminate components wrap the knob in a dedicated "Knob Centerer" frame - same footprint as the track, `layoutMode: HORIZONTAL`, `primaryAxisAlignItems`/`counterAxisAlignItems: CENTER`. ### 5.1 Thumb color bug fix (2026-07-21, same day)

Follow-up bug report: after آ§5's theme override shipped, the thumb rendered dark/black on every colored + checked variant instead of white. Root cause: `MuiSwitch-thumb`'s `backgroundColor` was left on stock MUI's `currentColor` default, and `switchBase`'s own `color` property (set per-color for the ripple/focus color, same convention MUI itself uses) was bleeding into the thumb fill via inheritance - the named color was "winning" the thumb, when only the track should carry it.

Fix, scoped to `src/theme/brandTheme.ts`:

- Removed the `color` property from `switchBase`'s base and checked rules entirely, since that's what was propagating into the thumb via `currentColor`.
- `thumb.backgroundColor` hardcoded to `'#ffffff'` unconditionally as the base fill - matches the Figma reference, where the thumb is pure white for Off and every On color variant.
- Preserved the one real exception confirmed against the Figma component: `color="default"` (the base/unnamed color) checked state has both track *and* thumb going near-black (grey/900), via a `'&.Mui-checked .MuiSwitch-thumb'` rule scoped inside `switchBase`.
- Each of the six named-color slots (`colorPrimary`/`colorSecondary`/`colorError`/`colorWarning`/`colorInfo`/`colorSuccess`) got a matching `'&.Mui-checked .MuiSwitch-thumb': { backgroundColor: '#ffffff' }` rule added alongside their existing checked-track rule, so the thumb reliably resolves back to white for every named color. This relies on MUI's own `overridesResolver` order (`ownerState.color !== 'default'` appends the `colorX` slot's styles *after* root/base styles), the same mechanism stock `Switch.js` uses - not a specificity hack.
- `IndeterminateSwatch` (web page, simulated preview) thumb fill updated from grey/900 to `'#ffffff'` to match; the "Anatomy & Token Specs" `SpecRow` copy updated to state the white-thumb-except-default rule explicitly.

**Verification**: same rsync-to-`/tmp` build workaround as آ§4/آ§5. Clean `next build`, 0 TypeScript errors, all 25 routes prerender. Ran `next start` and `curl`'d the built Switcher page, then inspected the raw emitted CSS directly (not just a successful compile): for every named-color instance, both the base `.Mui-checked .MuiSwitch-thumb{background-color:#212121;}` rule and the color-slot `.Mui-checked .MuiSwitch-thumb{background-color:#ffffff;}` rule appear under the *same* Emotion-generated class, with the `#ffffff` rule written after the `#212121` one in source order - confirming the white rule wins the cascade (equal specificity, later declaration wins). ### 5.5 Off/Indeterminate lightness flip (2026-07-22)

User request: flip which of Off vs Indeterminate reads as the lighter tone. Rationale: Indeterminate is a pre-interaction, "not yet decided" state - a switch can be toggled to Indeterminate only before the user ever interacts with it, and can never revert to it afterward, so it should read as the quieter/lighter of the two. Off is a real, determinate choice the user has made (or the default resting state), and should read as slightly more present than that. Previously Off was the lighter one (`grey/300`, `#e0e0e0`) and Indeterminate the darker one (`grey/400`, `#bdbdbd`) - the reverse of that intent.

This also drops the indirection through the shared "Disabled surface" semantic token, which "Off" was only ever borrowing coincidentally - Off and Disabled are different concepts, and `Semantic/Surface/Disabled` itself is untouched for whatever else references it.
- `components/switch/slideFillIndeterminate` (Indeterminate track): repointed from `grey/400` to `grey/300`.

Because every Off-track rectangle (all 7 colors أ— both sizes) and both Indeterminate-track rectangles were already bound to these two tokens by name, the flip propagated automatically to every variant and to the `Switcher â€” Documentation` canvas (live instances) with no per-node edits. The canvas's inline note next to the Indeterminate row was also rewritten to describe the new, correct rationale instead of the old (now backwards) "darker than a washed-out grey, lighter than Off" text.

Web changes: `MuiSwitch.styleOverrides.track.backgroundColor` (`brandTheme.ts`) changed from `baseTheme.palette.surface.disabled` to `colors.grey[400]` directly; `IndeterminateSwatch`'s (`page.tsx`) `bgcolor` changed from `grey[400]` to `grey[300]`.
